## Caption (front-loaded)

Voice-only AI is not accessible. I keep seeing demos pitched as accessibility wins because they replace typing with speech.

That is one user need. It is not the test.

WCAG 2.2 has a clear position. An interface accessible to speech but not to keyboard, focus-visible navigation, and screen readers fails 2.1.1, 2.4.7, and 4.1.2. Day one fail.

Why the myth keeps returning. Voice-as-accessibility has a real grain of truth. For motor disabilities, dyslexia, or temporary injury, speaking is genuinely easier than typing. Voice expands access for some.

But "expands access for some" is not the same as "is accessible."

Deaf users cannot use voice. Users in noisy environments cannot use voice. Users with speech disabilities or non-native pronunciation often cannot use voice reliably. Blind keyboard users need focus indicators and ARIA semantics to operate the surrounding UI.

The fix is not hard. It is just not a feature you can demo as a magic moment.

Build the voice flow. Then add a text input that does the same thing. Make the entire UI keyboard navigable. Surface the conversation as a focusable, screen-reader-readable transcript. Add visible focus on every interactive element. Caption every audio response.

You ship the voice demo and the WCAG-compliant version of the same flow. Both. Neither is optional.

GAAD week is a good moment to retire the framing. Voice as accessibility is half the story. Multimodal is the whole one.

Save this for the next AI demo someone calls accessible.
