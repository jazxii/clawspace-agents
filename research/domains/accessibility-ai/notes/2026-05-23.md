### DOJ ADA Title II IFR (2026-04-20) — Six Weeks On: Compliance Clock Reset, Obligations Not Suspended

- **Source**: [Federal Register — Extension of Compliance Dates IFR](https://www.federalregister.gov/documents/2026/04/20/2026-07663/extension-of-compliance-dates-for-nondiscrimination-on-the-basis-of-disability-accessibility-of-web) | [Jackson Lewis](https://www.jacksonlewis.com/insights/doj-extends-public-entities-compliance-deadline-ada-related-website-accessibility-hhss-may-2026-deadline-still-looms) | [Duane Morris alert](https://www.duanemorris.com/alerts/doj_extends_ada_title_ii_digital_accessibility_deadlines_one_year_0426.html)
- **Summary**: The DOJ's Interim Final Rule effective 2026-04-20 extends Title II web/mobile compliance for state/local entities ≥50,000 population from 2026-04-24 to 2027-04-26, and for entities <50,000 (and special districts) from 2027-04-26 to 2028-04-26. Public comments open until 2026-06-22. Critically, the IFR only delays the WCAG 2.1 AA compliance dates — it does NOT suspend Title II's nondiscrimination and effective-communication obligations, which have long supported web accessibility claims. HHS's separate May 11, 2026 healthcare deadline is unaffected by the DOJ IFR. AAPD issued a critical statement on the IFR; disability advocacy groups are mobilizing comments.
- **Quote**: "The interim final rule only delays the WCAG 2.1 Level AA compliance dates. It does not suspend Title II's underlying nondiscrimination and effective-communication obligations."
- **Why it matters**: Closes the "are active complaints being paused" question from the May 7 staged NotebookLM prompt — answer is no. Teams cannot use the extension as a reason to deprioritize remediation. Content angle: counter-narrative to "we got an extra year" framing.

### ACM Web for All 2026 Preprint Surfaced — LLM Barriers Paper Now Readable

- **Source**: [ResearchGate preprint](https://www.researchgate.net/publication/396615736_When_LLM-Generated_Code_Perpetuates_User_Interface_Accessibility_Barriers_How_Can_We_Break_the_Cycle) | [ACM DL](https://dl.acm.org/doi/10.1145/3744257.3744266) | [Preprints.org — Sustainable Web Accessibility w/ LLMs](https://www.preprints.org/manuscript/202508.0453)
- **Summary**: The Web for All 2026 paper "When LLM-Generated Code Perpetuates User Interface Accessibility Barriers, How Can We Break the Cycle?" is now available as a ResearchGate preprint, closing the 2026-05-08 open thread. Methodology: comparison of ChatGPT and Claude across accessibility-agnostic and accessibility-oriented prompts, evaluated against WCAG 2.1 criteria. Finding: accessibility-oriented prompts measurably increase success counts and reduce violation rates, but persistent barriers remain — particularly in semantic structure. Intervention point identified: prompt engineering can shift outcomes but cannot eliminate them.
- **Quote**: "Accessibility-oriented prompts increase success counts and reduce violation rates in WCAG criteria, but persistent barriers remain, particularly in semantic structure."
- **Why it matters**: This is the empirical anchor the domain has needed since May 4 — concrete evidence that prompt-level intervention is necessary-but-not-sufficient. Validates the "vibe fixing" prompt template library idea (ideas-feed 2026-05-08) AND clarifies its ceiling: prompts alone won't fix semantic structure issues; downstream remediation is still required.

### WebAccessBench (Feb 2026) — First Peer-Reviewed Cross-Model Digital Accessibility Benchmark

- **Source**: [WebAccessBench whitepaper](https://conesible.de/wab/whitepaper_webaccessbench.pdf) | [arXiv:2509.18965 — PDF Accessibility Benchmark](https://arxiv.org/abs/2509.18965)
- **Summary**: WebAccessBench (generated 2026-02-20) evaluates 19 LLMs on digital accessibility reliability across three guidance conditions (Unguided, Little, Expert). All 19 models improve from Unguided to Little guidance on average error count; 18/19 improve from Unguided to Expert. Separately, arXiv 2509.18965 ("Benchmarking PDF Accessibility Evaluation") introduces a scholarly PDF benchmark with expert-validated annotations across seven criteria including alt-text quality; GPT-4-Turbo led at 0.85 overall accuracy, but all models struggled with "Not Present" and "Cannot Tell" labels — especially for alt-text quality.
- **Quote**: "All 19 models improve from Unguided to Little guidance on average error count, and 18/19 improve from Unguided to Expert guidance." — WebAccessBench
- **Why it matters**: Provides the first multi-model, multi-guidance dataset directly applicable to the alt-text benchmark harness idea (ideas-feed 2026-05-04). Methodology validates the "rubric + corpus + multiple models" structure already in the harness spec; the GPT-4-Turbo 0.85 ceiling and the "Not Present / Cannot Tell" failure modes are the specific gaps a new harness could target.

### a11yscout (April 2026 launch) — axe-core-Wrapped GitHub Action with Source-Mapped Reporting

- **Source**: [Dev|Journal — a11yscout automation post (2026-04-24)](https://earezki.com/ai-news/2026-04-24-audit-wcag-21-accessibility-on-every-pull-request-free-github-action/)
- **Summary**: a11yscout is a GitHub Action that wraps axe-core's rule engine (not a separate scanner) and adds three layers: (1) PR-blocking semantics — fails builds on serious WCAG 2.1 violations; (2) source-mapping via a custom Vite plugin — reports errors at `src/components/Button.tsx:12:4` rather than CSS selectors; (3) automated PR comments per rule (e.g., `button-name` SC 4.1.2 A, `image-alt` SC 1.1.1 A). Closes the 2026-05-07 staged prompt on rule coverage — answer is "same as axe-core, plus source-mapping and PR-block semantics," not a different rule subset.
- **Quote**: "Source-mapping via a dedicated Vite plugin allows the scanner to report errors at the source file level instead of generic CSS selectors."
- **Why it matters**: The source-mapping layer is the actual innovation — for AI-generated code workflows where the developer needs to know which file/line a generated violation lives in, this dramatically reduces remediation time. Confirms the a11yscout benchmarking idea (ideas-feed 2026-05-07) is comparing PR-workflow ergonomics, not rule sets.
