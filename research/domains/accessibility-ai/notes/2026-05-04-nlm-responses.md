## Q: What WCAG success criteria are most frequently violated in v0 / Bolt / Lovable-generated React apps?

**A (grounded from 2026-05-02 notes, section 3 "Cognitive load is multiplied by AI-generated UI"):**

Based on teardown analysis documented in the domain notes (sourced from Deque's 2025 "Auditing AI-generated UI" series and WebAIM analyses of Bolt-generated React apps), the most frequent violation clusters in AI-generated React apps are:

1. **SC 3.2.6 Consistent Help (A)** — Help links and contact mechanisms appear in inconsistent positions across pages. Generated apps have no awareness of cross-page consistency; each page is independently scaffolded.

2. **SC 3.3.8 Accessible Authentication (AA)** — Generated apps default to email + password + email-confirm-code flows. If the OTP cannot be pasted (clipboard blocked), this fails 3.3.8. Passkeys and WebAuthn are almost never generated by default.

3. **SC 1.4.11 Non-text Contrast (AA)** — Decoration density (gradients, low-contrast icons, background images) routinely produces UI components below the 3:1 contrast threshold required for UI components and graphical objects.

4. **SC 4.1.3 Status Messages (AA)** — Loading, success, and error states in generated UIs often rely on brief animations. Without live regions (`role="status"` or `role="alert"`), these state changes are invisible to screen reader users.

5. **SC 2.4.3 Focus Order (A) / SC 2.1.1 Keyboard (A)** — Custom interactive elements built with divs and spans rather than semantic HTML lack keyboard operability and focus management by default.

6. **SC 1.4.1 Use of Color (A) / SC 3.3.1 Error Identification (A)** — Form validation errors in generated UIs frequently rely on red border styling alone, with no programmatic text description of the error.

The PRD also identifies auditor tooling (axe-core, Pa11y, IBM Equal Access) as an area to watch for scanner results against AI-generated output.

From the domain framing and general accessibility knowledge consistent with the domain's sources (Deque blog, Adrian Roselli, TPGi blog), the highest-risk ARIA misuse patterns in AI-generated component code include:

- **Dialog/modal patterns**: `role="dialog"` added without `aria-modal="true"`, missing `aria-labelledby`, and no focus trap implementation. LLMs commonly scaffold visually correct modals that are functionally inaccessible.
- **Combobox patterns**: The APG combobox pattern requires careful coordination of `aria-expanded`, `aria-activedescendant`, and `aria-controls`. LLMs frequently generate partial implementations.
- **Tab patterns**: `role="tablist"` / `role="tab"` / `role="tabpanel"` triads are commonly generated without the required `aria-selected` state management or arrow-key navigation.
- **Live regions**: `aria-live` regions added but with incorrect politeness levels (`assertive` where `polite` is correct), causing interrupt storms.

## Q: How does cognitive load (per COGA) shift when LLM-driven UIs use dynamic layout vs. static templates?

**A (grounded from 2026-05-02 notes, sections 3 and 4):**

This is directly addressed in the domain's existing research notes (2026-05-02-cognitive-load-neurodiversity.md), grounded from W3C COGA "Making Content Usable" and the Deque 2025 AI-generated UI audit series.

Key findings:

**Dynamic layout increases cognitive load for neurodivergent users across multiple COGA dimensions:**

1. **COGA Objective 6 ("Ensure processes do not rely on memory")** — Dynamic layouts shift the burden of spatial memory onto the user. When navigation, action buttons, or content blocks reflow between page loads or states, users with memory or attention differences must re-learn the interface on each interaction. Static templates externalise that structure so working memory is freed for the task.

2. **COGA Objective 1 ("Help users understand what things are")** — Consistent affordances (same button style = same action type) are broken when LLM-driven UIs generate novel component styling per page or per session. Users with cognitive disabilities rely on learned patterns more than sighted, neurotypical users.

3. **ADHD-specific impact** — Predictability matters more than novelty for ADHD users. Surprise modals, autoplaying carousels, and layout shifts (even CSS-driven) are documented as hostile patterns. The existing notes cite: "Predictability matters more than novelty. Surprise modals, autoplaying carousels, and 'shortcut' gestures are hostile."

4. **Autism-specific impact** — "Features that change position between sessions (A/B tests on logged-in users) are particularly painful" per the domain notes, citing National Autistic Society digital guidance.

5. **SC 3.2.6 Consistent Help** — AI-generated multi-page apps rarely maintain help-mechanism position consistency. Each page is scaffolded independently, violating the predictability requirement.

**Static templates reduce cognitive load** by converting spatial memory tasks into recognition tasks — a well-established principle from cognitive psychology (recognition vs. recall, Lidwell et al.). COGA's design objective 6 ("visible state, progress, recovery") is far easier to satisfy with static templates where state can be predictably surfaced.

## Q: What does WCAG 2.2 SC 3.3.8 (Accessible Authentication, Minimum) explicitly require, and which auth patterns satisfy it without relying on cognitive function tests?

**A (grounded from 2026-05-02 notes, section 1 "WCAG 2.2 cognitive Success Criteria"):**

**What SC 3.3.8 requires (AA, Level AA):**

SC 3.3.8 states that at least one method of authentication in any authentication process must not require a cognitive function test, unless an alternative is provided that does not rely on a cognitive function test.

A "cognitive function test" is defined as a task that requires the user to remember, manipulate, or transcribe information — examples include:
- Solving a puzzle (image CAPTCHA, drag-to-match)
- Memorising or transcribing a character string without being able to paste it
- Answering a knowledge-based question ("What was your first pet's name?")

**Auth patterns that satisfy SC 3.3.8:**

1. **Passkeys / WebAuthn** — biometric or hardware-key authentication with no memory or transcription requirement. Fully satisfies 3.3.8.

2. **Password manager + paste-enabled password field** — if the user can autofill or paste their password (i.e., the field does not block clipboard), the cognitive transcription burden is eliminated. Password managers handle recall; the user need only authenticate the manager.

3. **Copy-paste OTP** — one-time password sent by email or SMS that the user can paste (not manually transcribe). Many implementations block paste; these fail 3.3.8 in spirit. Allowing paste satisfies it.

4. **OAuth / SSO ("Sign in with Google/Apple")** — delegates authentication to an already-authenticated session; no new cognitive function test required.

5. **Magic link** — one-click link sent to email; no transcription. Satisfies 3.3.8 provided the link destination does not then require a cognitive test.

**Patterns that fail SC 3.3.8:**

- Image CAPTCHA requiring object identification
- CAPTCHA requiring character transcription without audio alternative
- Email OTP where the field blocks paste
- Time-limited code with no copy button and no extension mechanism

The domain notes also flag: "Time-limited password reset code with no copy button — fails 3.3.8 in spirit; use passkeys instead."

SC 3.3.9 (AAA) extends the same requirement to all authentication steps, not just one.

## Q: What concrete practices from W3C COGA "Making Content Usable" are most often missing in AI-generated UIs (v0, Bolt, Lovable)?

**A (grounded from 2026-05-02 notes, sections 2, 3, and 8 "Common failure modes"):**

W3C COGA "Making Content Usable" defines 8 design objectives. Based on the domain notes' teardown analysis of AI-generated UIs:

**Most frequently absent COGA practices:**

**Objective 1 — Help users understand what things are and how to use them**
- AI-generated UIs inconsistently label interactive elements. Icon buttons lack visible text labels. Tooltips are often absent or only CSS-powered (inaccessible). Generated components use novel visual affordances without established conventions.

**Objective 2 — Help users find what they need**
- Multi-page generated apps do not maintain consistent navigation structure. Breadcrumbs are rarely generated. Search is frequently absent.

**Objective 3 — Use clear and understandable content**
- Generated UX copy (button labels, error messages, placeholder text) varies in register and reading level. Marketing-inflected copy ("let's get started!", "almost there!") is not plain language. Error messages are often technical (console-adjacent strings).

**Objective 4 — Help users avoid mistakes and know how to correct them**
- Form validation is commonly implemented with only visual indicators (red border). Text descriptions of errors are frequently missing or detached from the field (fails SC 3.3.1, 3.3.3).

**Objective 5 — Help users focus**
- Visual noise is the defining characteristic of AI-generated UIs: decorative icons, gradient backgrounds, animated elements, autoplaying hero sections. None of these are removed or made optional.

**Objective 6 — Ensure processes do not rely on memory**
- Generated multi-step forms do not persist data between steps. No progress indicators are generated by default. No visible state is maintained on back-navigation.

**Objective 7 — Provide help and support**
- Help mechanisms are almost never generated unless explicitly prompted. No "contact us" link, no inline guidance, no progressive disclosure of instructions.

**Objective 8 — Support adaptation and personalization**
- Generated UIs rarely respect `prefers-reduced-motion`, `prefers-color-scheme`, or user font-size preferences. Custom CSS that overrides browser defaults without providing equivalent mechanism fails objective 8.

The domain notes specifically flag: "Auth flows: Generated apps default to email + password + email-confirm-code dance — fails 3.3.8 unless the OTP can be pasted" and "Loading, success, and error states often only animate briefly; users with slower processing miss them."

## Q: What are the most cited UX patterns for ADHD-friendly form design (re-entry, autosave, progress indication)?

**A (grounded from 2026-05-02 notes, section 4 "ADHD" and section 3 "Cognitive load multiplied by AI-generated UI"):**

The domain notes document the following ADHD-specific form design patterns, grounded from UK GDS, ADHD UK web design notes, and the COGA "Making Content Usable" guidance:

**1. Autosave / draft persistence**
- ADHD users experience task interruption at higher rates due to attention switching. Forms that lose data on interruption impose disproportionate cognitive penalty. The "save-as-you-go" pattern is cited as one of the strongest curb-cut effects in the notes: "helps ADHD users with task interruption, helps everyone with a flaky connection."
- Implementation: periodic silent autosave to localStorage or server, with visible "saved" confirmation (using a live region so screen readers also confirm).

**2. Progress indication**
- Multi-step forms must show visible progress (step N of M) at all times, not just at the top of the current step. COGA Objective 6 ("visible state, progress, recovery") requires this.
- The notes document: "every additional required field after step 3 measurably increases abandonment" — progress indication mitigates this by making the remaining load visible.

**3. Chunking / step-gating**
- Break long forms into steps of 3-5 fields maximum. This limits the working-memory demand of any single view.

**4. Re-entry tolerance (no data loss on back-navigation)**
- The "back" button must not clear fields already completed. This is rarely implemented by default in AI-generated forms and is a documented failure mode (COGA Objective 6).

**5. No time pressure**
- Remove countdown timers from form contexts. WCAG 2.2.1 (Timing Adjustable) is the floor; ADHD UK guidance recommends eliminating artificial timers entirely. The notes state: "Time-pressure UI (countdown timers, 'your seat is held for 4:59') spikes cortisol and triggers task-switching failures."

**6. Plain, predictable labels**
- Labels above fields (not inside as floating labels that disappear), consistent casing, and no clever microcopy that obscures the field's purpose. Idioms in field instructions ("ballpark figure", "quick as possible") add parse cost.

**7. Error recovery without restart**
- Validation errors must be surfaced inline, adjacent to the field, with a clear corrective instruction. Users should never have to restart a form due to a validation error. Focus should move to the first erroneous field (or an error summary) on submission attempt.

## Q: How do dyslexia-friendly typography recommendations (line-height, line-length, font choice) hold up under recent empirical scrutiny?

**A (grounded from 2026-05-02 notes, section 4 "Dyslexia" and section 5 "Plain language"):**

The domain notes document the following, grounded from the British Dyslexia Association style guide (2023) and supporting research:

**Line-height — strong evidence**
- Increased line-height (1.5 or greater, relative to font size) is consistently recommended and empirically supported. The notes state: "Font choice matters less than spacing. Increased line-height (1.5+), letter-spacing, and shorter line lengths (45-75ch) help most." This aligns with WCAG 1.4.12 (Text Spacing) which sets a minimum of 1.5x line-height as a no-loss-of-content threshold.

**Line-length — moderate evidence**
- The 45-75 character range is supported for dyslexic readers. Excessively long lines (greater than 80ch) increase tracking difficulty and the probability of losing one's place on return sweep. The evidence base is stronger for this recommendation than for font choice.

**Letter-spacing — moderate evidence**
- Slightly increased letter-spacing (0.05-0.12em above default) is recommended by BDA. WCAG 1.4.12 also covers letter-spacing (minimum 0.12x font-size without loss of content).

**Font choice — weak/contested evidence**
- The notes explicitly state: "Serif vs. sans-serif evidence is weaker than commonly claimed." Specialised dyslexia fonts (OpenDyslexic, Sylexiad) have been tested in multiple studies with mixed results. Several RCTs have found no statistically significant advantage over standard sans-serif fonts (e.g., Arial, Verdana) at matched size and spacing. The mechanism claimed (weighted letterforms reducing rotation confusion) has not been consistently demonstrated.
- The current evidence-based recommendation is: use a clean sans-serif, avoid decorative fonts, and prioritise spacing over typeface choice. Do not mandate proprietary dyslexia fonts in public-facing design — allow user override instead.

**Justified text — consistent evidence against**
- Left-aligned text is consistently preferred over justified text for dyslexic readers. Justification creates uneven word spacing ("rivers") that disrupts the visual rhythm of return sweeps.

**Reading mode / text-to-speech — strongest preference signal**
- "Reading mode + text-to-speech are the most-requested adaptations" per the notes. These are functional adaptations that transcend typography: they shift the presentation modality rather than tuning the visual parameters.

**Summary for practitioners:**
Prioritise: line-height 1.5 or greater, line-length 45-75ch, left alignment, no justified text, user control over font size. Deprioritise: proprietary dyslexia fonts (insufficient evidence). Enable: text-to-speech and reading mode as complementary adaptations.

**Citations:**
- [British Dyslexia Association Dyslexia-Friendly Style Guide (2023)](https://www.bdadyslexia.org.uk/advice/employers/creating-a-dyslexia-friendly-workplace/dyslexia-friendly-style-guide)
- [W3C WCAG 2.2 — SC 1.4.12 Text Spacing](https://www.w3.org/TR/WCAG22/#text-spacing)
