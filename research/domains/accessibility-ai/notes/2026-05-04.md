# Research notes — accessibility-ai — 2026-05-04

## Summary

The WCAG 3.0 contrast algorithm is officially back to square one — APCA was dropped in 2023 and the April 2026 W3C editor's draft confirms no replacement has been chosen, with WCAG 3.0 likely no earlier than 2030. Meanwhile the WebAIM Million 2026 report finds accessibility errors on 95.9% of top sites (3.9% of all page elements), with page complexity growing 22.5% annually and outpacing remediation. On the tooling side, axe-core 4.11.x ships AI-assisted guided testing covering an average 57% of WCAG failures automatically, and axe Platform now supports France's RGAA standard. For native HTML patterns, the dialog+popover pair have reached Baseline status across browsers, making custom ARIA focus-trap implementations largely redundant. Alt-text quality benchmarking against frontier LLMs remains an open research gap with no peer-reviewed head-to-head published yet in 2026.

## Items

### WCAG 3.0 Contrast Algorithm: Still Undetermined as of April 2026

- **Source**: [Adrian Roselli — adrianroselli.com](https://adrianroselli.com/2026/04/wcag3-contrast-as-of-april-2026.html)
- **Summary**: The W3C April 8, 2026 editor's draft for WCAG 3.0 explicitly states the contrast algorithm "is yet to be determined." APCA was removed from the draft in mid-2023 after failing to gain working-group support.
- **Quote**: "WCAG3 is years away from being done...perhaps 2030 at the soonest."
- **Why it matters**: Any team that has been experimenting with APCA ratios for AI-generated color palettes or design-token contrast checks needs to revert to WCAG 2 requirements — there is no sanctioned alternative until the standard finalizes.

### WebAIM Million 2026 — Persistent Failure Categories and AI-Generated UI Risk

- **Source**: [D2i Technology — Accessibility Testing 2026](https://d2itechnology.com/blogs/accessibility-testing-2026-wcag-3-ada-ai-compliance/)
- **Summary**: The 2026 WebAIM Million report finds accessibility errors present in 3.9% of all page elements across the top million sites. Page complexity is growing 22.5% annually, creating new failure points that standard automated scans cannot catch. The top four failure categories are missing alt text, insufficient color contrast, unlabelled form fields, and empty link text — all categories that AI-generated UI tools reproduce at scale.
- **Quote**: "Automated tools — even sophisticated AI-assisted ones — typically catch between 30% and 40% of WCAG failures."
- **Why it matters**: Directly quantifies the accessibility debt AI-generated UI tools impose and establishes the baseline that any LLM-UI pairing must beat to claim accessibility parity.

### WCAG 3.0 March 2026 Working Draft — Scoring Approach Confirmed Non-Binding

- **Source**: [D2i Technology — Accessibility Testing 2026](https://d2itechnology.com/blogs/accessibility-testing-2026-wcag-3-ada-ai-compliance/)
- **Summary**: The March 2026 WCAG 3.0 working draft introduces a scoring-based model (not pass/fail) and expands technology coverage to native mobile and AI-generated content, but the W3C explicitly states it "should not be used as a compliance target."
- **Quote**: "It is not yet a recommendation standard — meaning it should not be used as a compliance target."
- **Why it matters**: Confirms WCAG 2.2 AA remains the only legally defensible standard for any AI-generated content workflow through at least 2030; scoring-based WCAG 3.0 framing is useful for internal maturity conversations but not for contracts or litigation defense.

### Native dialog + popover Reach Baseline — Custom ARIA Focus-Trap Implementations Largely Redundant

- **Source**: [web.dev — dialog and popover: Baseline layered UI patterns](https://web.dev/articles/baseline-in-action-dialog-popover)
- **Summary**: Both the HTML `<dialog>` element and the Popover API are now Baseline across modern browsers. `showModal()` automatically applies `aria-modal="true"`, handles focus trapping, and restores focus on close. The Popover API handles `aria-expanded`, `aria-popup`, and `aria-controls` natively without custom JS. The two differ critically: dialogs make background content inert; popovers do not.
- **Quote**: "You get all of the web features just described that are now Baseline" — eliminating the need for extensive custom logic.
- **Why it matters**: AI-generated UI tools (v0, Bolt, Lovable) that still scaffold `<div role="dialog">` with hand-rolled focus-trap scripts are producing WCAG 4.1.2 violations that a single native element swap would fix — this is the single highest-leverage ARIA pattern to flag in generated code reviews.

### APG Dialog Pattern — Current State (May 2026)

- **Source**: [WAI ARIA Authoring Practices Guide — Dialog Modal Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/)
- **Summary**: The APG dialog pattern still specifies `aria-modal="true"` on the container, focus moved to an element inside on open, keyboard focus constrained within until close, and focus returned to the triggering element on close. The `showModal()` method now satisfies all of these automatically in Baseline-compliant browsers, making the ARIA pattern a fallback specification rather than an implementation guide for modern stacks.
- **Quote**: "Using the showModal() method adds the dialog to the top layer, implicitly adds the aria-modal='true' attribute, and automatically makes all of the elements on the page underneath the dialog inert."
- **Why it matters**: Teams and AI tools implementing ARIA manually without using native `<dialog>` are adding maintenance burden and introducing WCAG 4.1.2 / 2.1.2 failure risk — knowing the APG and the native-HTML equivalent are now aligned removes the "which do I follow?" ambiguity.
